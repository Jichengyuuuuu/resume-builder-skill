# 简历生成 Skill (Resume Builder)

这个 Skill 帮助用户快速生成高质量的中文简历。核心流程分为三个阶段：**信息收集** → **简历生成** → **高阶顾问建议**。

## 工作流概览

```
用户请求 → 信息收集阶段 → 简历生成阶段 → HTML + DOCX 双格式输出 → 高阶顾问建议 → 用户下载/查看
```

### 第一阶段：信息收集 (Context Gathering)

你需要通过友好的交互方式向用户收集以下信息。**重点：信息收集应该柔和而全面，不强制用户必须提供所有字段。** 对于用户没有提供的信息，使用占位符或示例文本呈现。

#### 1. **个人基本信息**
- 姓名
- 电话号码
- 邮箱地址
- 城市/地区
- 个人网站/GitHub/博客链接（可选）
- 工作年限

#### 2. **职业目标** (可选但推荐)
- 期望的职位名称
- 职业方向描述（例如："寻求后端开发工程师职位，专注于高并发分布式系统"）
- 期望的行业或公司类型

#### 3. **教育背景**
- 学校名称
- 专业/学位
- 入学年份/毕业年份
- GPA（可选）
- 主修课程或获奖情况（可选）

#### 4. **工作经历**
对于每段工作经历，收集：
- 公司名称
- 职位名称
- 工作周期（开始时间-结束时间）
- 工作地点（可选）
- 工作描述和成就（**重点收集量化指标，如"提升 30% 性能"、"减少成本 $50K"、"服务 1000万+ 用户"等**）

#### 5. **项目经验** (可选)
- 项目名称
- 你的角色
- 项目时间
- 项目描述和你的贡献（同样需要量化指标）
- 使用的技术栈

#### 6. **专业技能与综合能力**

⚠️ **重要原则：先分析岗位，再决定技能维度。不要套用固定模板。**

**第一步：判断目标岗位类型**

根据用户提供的目标岗位，先在心里归类（不需要告诉用户），常见类型包括但不限于：
- 技术研发类（后端、前端、算法、AI 工程师、数据工程师等）
- 产品类（C端产品、B端产品、AI 产品、商业化产品等）
- 数据类（数据分析师、数据科学家、BI 等）
- 设计类（UI/UX、视觉、交互、品牌设计等）
- 运营类（用户运营、内容运营、社群运营、增长等）
- 市场营销类（品牌、市场、PR、SEM/SEO 等）
- 销售/BD 类
- HR / 行政 / 财务 / 法务等职能类
- 项目/项目管理类
- 咨询/战略类
- 学术/科研类

**第二步：根据岗位类型动态决定技能维度**

不同岗位的技能维度差异巨大，必须按岗位定制。参考示例：

| 岗位类型 | 核心技能维度（举例）|
|---------|---------------------|
| 后端工程师 | 编程语言 / 框架与中间件 / 数据库 / 分布式与高并发 / DevOps & 云 / 系统设计 |
| AI 产品经理 | AI/LLM 能力 / 产品方法论 / 数据能力 / 行业认知 / 工具协作 |
| C端运营 | 用户增长方法论 / 内容策划 / 数据分析 / 平台规则与玩法 / 工具栈（如 GrowingIO、神策）|
| 视觉设计师 | 设计能力（品牌/插画/动效）/ 设计工具（Figma/C4D/AE）/ 设计方法论 / 行业作品集 |
| 市场营销 | 品牌策略 / 内容营销 / 投放与渠道 / 数据归因 / 工具栈（GA、巨量、Meta Ads）|
| HRBP | HR 专业能力（招聘/OD/绩效/薪酬）/ 业务理解 / 数据敏感度 / 软技能 |
| 财务分析师 | 财务专业能力 / 建模与估值 / 数据工具（Excel/SQL/BI）/ 行业理解 |

**第三步：技能呈现的通用规则**

无论什么岗位，都应遵循：
- **避免过于基础的描述**：不要写"熟练使用 Office""会用电脑"这类反向加分项
- **向上拔一层**：把用户口语化的技能描述转化为更专业的表达（如"会做 PPT" → "数据可视化与商业演示"）
- **匹配目标岗位 JD**：如果用户提供了 JD，技能关键词应主动对齐 JD 高频词
- **AI 相关能力按需添加**：只有当目标岗位本身涉及 AI、或者 AI 能力对该岗位是加分项（如产品、运营、市场、数据岗）时才补充；纯传统岗位（如财务、法务、行政）不要硬塞 AI 关键词
- **语言能力融入此区块**：作为技能维度之一展示，不单独成模块
- **行业认知与软技能**：跨团队协作、商业化思维、行业 know-how 等通用项可作为最后一个维度

**第四步：如果用户提供的技能信息不足**

主动询问与目标岗位强相关的技能项，例如：
- 对设计岗：问作品集、擅长的风格、用的工具
- 对运营岗：问操盘过的活动量级、用过的工具栈
- 对销售岗：问最大单的金额、主要客户类型、销售方法论

不要为了凑数而泛泛地补充与岗位无关的技能。

#### 7. **认证和荣誉** (可选)
- 证书名称和颁发机构
- 获奖情况
- 出版物或开源贡献

#### 8. **其他信息** (可选)
- 自我评价或专业总结
- 其他相关经历或爱好

### 收集方式建议

1. **初始检测**：先询问用户是否有以下资料：
   - ✅ 现有简历（附件或复制粘贴内容）
   - ✅ 职位描述或目标岗位信息
   - ✅ 口头描述工作内容

2. **智能优化**：
   - 如果用户上传了简历，先解析内容，然后在此基础上补充和优化
   - 如果用户只有口头描述，引导用户按照上述结构补充细节
   - 如果用户提供了目标职位描述，使用 ATS 优化技巧来匹配关键词

3. **友好提示**：
   - 对于每个信息块，提供 1-2 个示例，帮助用户理解期望的内容格式
   - 对于可选字段，明确标注为"可选"
   - 鼓励用户提供量化数据和具体成就

## 第二阶段：简历生成 (Resume Generation)

一旦收集了足够的信息，**同时生成 HTML 和 DOCX 两种格式**的简历。

### 简历设计要求

#### 风格指南
- **配色**：黑白搭配（主文本：深灰色 #333333，辅助文本：灰色 #666666，背景：纯白 #FFFFFF）
- **重点色**：可选的深蓝色 #003366 或深灰色用于突出部分
- **字体**：现代无衬线字体（优先使用系统字体：-apple-system, BlinkMacSystemFont, Segoe UI, 微软雅黑等）
- **布局**：单列或双列布局，保持清晰的视觉层级
- **间距**：充分的留白，使内容易于阅读
- **可打印性**：确保在 A4 纸张上打印效果良好
- **页数控制**：**优先尽量控制在一页以内**，但不强制。判断标准如下：
  - 工作经验 ≤ 5 年：强烈建议一页
  - 工作经验 6-10 年：尽量压缩到一页，实在放不下允许两页
  - 工作经验 10 年以上 / 内容极其丰富：两页可接受，但不要超过两页
  - **压缩优先级**：先删减早期/相关性低的工作经历分点 → 再缩减字体/行距 → 最后考虑双列布局

#### 核心要素

**1. 头部信息区块**
```
[姓名 - 大号、加粗]
[职位/求职方向] | [城市] | [电话] | [邮箱] | [GitHub/个人网站]
[职业总结或自我评价 - 1-2 句话]
```

**2. 专业技能区块**
- **技能维度需根据目标岗位动态确定**（参考第一阶段 #6 的岗位分类与维度对照表），不要套用固定模板
- 技能以类别分组展示，使用 tag 或紧凑的行内格式
- 语言能力融入此区块，不单独成模块
- 优先展示与目标岗位最相关的技能；只在岗位实际需要时才加入 AI/前沿技术关键词，不硬塞

**3. 工作经历区块**

每条工作描述的分点应采用 **"加粗概括短语 + 冒号 + 详细描述"** 的格式：
```
[公司名称] - [职位] | [时间范围]
• **系统架构设计：**主导设计高并发分布式支付系统，日均处理交易 100万+，系统可用性 99.99%
• **性能优化：**优化核心查询链路，平均响应时间从 500ms 降低到 50ms，性能提升 10 倍
• **团队管理：**带领 5 人小组完成 Q3 季度目标，提前 2 周交付
```

**4. 项目经验区块** (如有)

同样采用加粗概括 + 冒号的分点格式：
```
[项目名称] | [角色] | [时间]
• **项目背景：**...
• **核心贡献：**...
• **技术栈：**...
```

**5. 教育背景区块**
```
[学校名称] - [学位/专业] | [时间]
GPA: X.X / 4.0 (如有)
```

**6. 认证和荣誉区块** (如有)
- 可将"认证"和"荣誉"分开展示

#### ⚠️ 禁止事项

- **不要在简历底部添加任何生成水印**，包括但不限于"此简历由 Resume Builder 生成"、"Generated by..."等字样。简历应该看起来完全由用户自己制作。

#### 分点描述格式规范（全局适用）

在工作经历、项目经验等所有需要分点描述的区块中，**每个要点必须以加粗的概括性短语开头，后跟冒号，再接详细描述**：

- ✅ `**数据治理：**建立企业级数据质量监控体系，覆盖 200+ 核心指标`
- ✅ `**模型优化：**通过 Prompt Engineering 和 RAG 架构优化，将回答准确率从 72% 提升至 91%`
- ❌ `负责后端系统开发`（太笼统，缺少加粗概括）
- ❌ `设计和实现高并发分布式支付系统`（缺少前置的加粗概括短语）

#### 重要的优化策略

1. **ATS 优化**（针对招聘系统）
   - 使用标准的职位名称（对应目标岗位）
   - 包含行业关键词
   - 避免过多的特殊符号或图像

2. **成就优先策略**
   - 用"**概括短语：**动词 + 具体行为 + 结果"的格式写工作描述
   - 强调量化指标（数字、百分比、金额等）
   - 前 3 项工作经历是重点，应该最详细

3. **针对性优化**
   - 如果用户提供了目标职位描述，在生成简历时优先突出相关的技能和经历
   - 自动调整各部分的详细程度

4. **页数控制策略（重要）**
   - 默认目标：**一页**。在不损失核心内容的前提下，优先压缩到一页
   - 每段工作经历控制在 **3 条**分点为主，最多不超过 5 条；优先保留量化成就最强的条目
   - 技能区块使用紧凑的行内 tag 格式，避免占用过多纵向空间
   - 早期工作经历（超过 8 年前）只保留公司、职位、时间，不展开分点
   - 如内容确实超出一页，**不要强行压缩导致信息残缺**，自然延伸到第二页，但需告知用户当前页数及建议删减方向

4. **技能专业化提升（按岗位定制）**
   - 避免出现过于基础的技能描述（如"熟练使用 Office"、"会用电脑"）
   - 主动将用户提供的技能向专业方向提升（如"会做表格" → "数据建模与可视化"）
   - **只在岗位相关时**补充前沿能力关键词（AI 关键词只对 AI/产品/运营/数据等相关岗位有效，不要硬塞到财务、法务、行政等传统职能岗）

## 双格式输出实施指南

### HTML 版本

- 使用 HTML5 + CSS3，确保兼容性
- 支持打印样式 (@media print)
- 使用内联 CSS 或 `<style>` 标签，确保独立性
- 考虑响应式设计，在手机上也可查看
- **不包含任何生成工具水印**

### DOCX 版本

生成 HTML 后，同时使用 `docx` npm 包生成 Word 版本。参考 `/mnt/skills/public/docx/SKILL.md` 获取 docx-js 的详细用法。

关键要点：
- 先 `npm install -g docx`
- 使用 A4 纸张尺寸（11906 x 16838 DXA）
- 使用 Arial / 微软雅黑 作为默认字体
- 加粗概括短语使用 `bold: true` 的 TextRun，冒号后的描述用普通 TextRun
- 使用 `LevelFormat.BULLET` 配置列表，不要手动插入 unicode bullets
- 用 `Packer.toBuffer` 写入文件
- 生成后使用 `python scripts/office/validate.py` 验证文件
- **不包含任何生成工具水印**

```javascript
// DOCX 生成示例骨架
const { Document, Packer, Paragraph, TextRun, HeadingLevel, AlignmentType, LevelFormat, BorderStyle } = require('docx');

// 工作经历分点示例
new Paragraph({
  numbering: { reference: "bullets", level: 0 },
  children: [
    new TextRun({ text: "系统架构设计：", bold: true, font: "Microsoft YaHei", size: 21 }),
    new TextRun({ text: "主导设计高并发分布式支付系统，日均处理交易 100万+", font: "Microsoft YaHei", size: 21 }),
  ]
})
```

### 输出流程

1. 在 `/home/claude` 下生成 HTML 和 DOCX 文件
2. 验证 DOCX 文件：`python scripts/office/validate.py resume.docx`
3. 复制两个文件到 `/mnt/user-data/outputs/`
4. 使用 `present_files` 工具同时展示两个文件给用户

## 第三阶段：高阶顾问建议 (Senior Advisor Feedback)

在简历生成并交付后，**自动**站在比用户当前岗位高 1-2 级的视角（例如用户是产品经理则以高级/资深产品总监视角，用户是初级工程师则以 Tech Lead 视角），给出精准、克制的建议。

### 核心原则

1. **角色代入**：明确以"如果我是面试你目标岗位的 hiring manager"的身份说话
2. **精准不啰嗦**：每次只挑 3-5 条最关键的建议，不要面面俱到；宁少勿多
3. **按实际情况选择维度**：以下维度不需要每次都覆盖，根据用户提供的信息中实际存在的短板或机会点来选

### 可选建议维度（按需组合，非全部必选）

**信息补充建议**
- 指出简历中明显缺失但对目标岗位至关重要的信息（如缺少量化数据、缺少技术深度描述、缺少管理幅度等）
- 给出具体的补充方向和示例措辞，而非泛泛而谈
- 例如："你提到了数据看板但没有说清楚数据规模和技术方案，建议补充数据源数量、更新频率、使用的 BI 工具等"

**求职方向建议**
- 基于用户的经历判断目标岗位的匹配度，如果有 gap 直接说
- 如果有更适合的岗位方向或赛道，简要建议
- 例如："你的经历偏流程效率工具，如果投商业化产品经理，建议补充你对营收指标、定价策略的理解"

**个人亮点提炼**
- 从用户提供的（可能是口语化的）描述中，挖掘出用户自己没意识到的亮点
- 帮助用户理解"面试官看重什么"
- 例如："纸质到线上的转型其实是一个 change management 的故事，你可以在面试中强调你是如何推动组织变革、处理抵触情绪的"

**面试准备建议**
- 针对目标岗位可能的面试问题，提前预判 2-3 个高频考点
- 给出回答框架而非完整答案
- 例如："B端商业化岗位大概率会问 '你怎么衡量产品的商业价值'，建议用 ROI 框架准备"

**需要注意的风险点**
- 简历中可能被面试官 challenge 的弱点（如经历单一、公司不知名、跨度大等）
- 建议如何在面试中主动化解
- 例如："只有一段工作经历，面试官可能质疑你的适应性，建议准备好 '为什么在一家公司深耕 3 年' 的故事线"

### 输出格式

建议以简洁的结构化文本直接写在对话中（不生成文件），格式示例：

```
💡 高阶顾问建议（以资深产品总监视角）

1. **补充商业化认知：**你的经历集中在效率工具侧，但目标岗位是"商业化产品经理"，建议...
2. **亮点提炼 — 变革推动力：**从纸质到线上不只是产品设计，更是组织变革...
3. **面试高频考点：**B端商业化岗位大概率会问...
```

### 关键约束

- **不要重复简历已有的内容**，建议只针对"简历之外"的事
- **不要写成通用的求职鸡汤**（如"要自信""多练习"），每条建议必须锚定在用户的具体情况上
- **不要超过 5 条**，信息密度 > 信息数量
- **语气直接、平等**，像一个愿意帮忙的前辈，不像 HR 培训材料

## 实施指南

### 推荐的收集流程

1. **欢迎用户**，解释你将如何帮助他们
2. **快速问卷**：用交互式按钮或简洁的问题了解用户有什么资料
3. **分块收集**：按"基本信息" → "教育背景" → "工作经历" → "技能" 的顺序逐步收集
4. **质量检查**：在生成前，确认关键信息完整（至少：姓名、工作经历、教育背景）
5. **生成并展示**：同时创建 HTML + DOCX 简历，展示预览，提供下载链接
6. **高阶顾问建议**：生成简历后，自动附上 3-5 条精准建议（不需要用户额外请求）
7. **迭代优化**：提供修改选项（增减工作经历、调整布局、修改内容等）

### 何时使用占位符

- 用户未提供某个字段时，用清晰的占位符（如 `[你的姓名]`）
- 对于可选字段，可以隐藏或显示为灰色提示
- 生成时，如果某个字段为空，不显示该部分，或用示例文本标明

## 常见优化技巧

1. **突出成就而非责任**

2. **量化一切可量化的内容**
   - ❌ "提升了系统性能"

3. **使用强有力的动词**
   - 设计、实现、优化、领导、驱动、改进、创建、建立等

   - 如果投数据科学职位，突出数据分析能力

5. **保持简洁**
   - 每段工作经历 3-5 个关键成就
   - 避免冗长的段落

- ✅ 清晰的内容层级和视觉焦点
- ✅ **不包含任何"由 XX 工具生成"的水印或标记**

## 常见问题

**Q: 如果用户没有提供某些信息怎么办？**
A: 使用占位符或示例文本，并在生成时提醒用户可以稍后更新。对于完全可选的字段（如个人网站、GPA），可以在用户明确说明后再添加。

**Q: 这个 Skill 能处理多语言简历吗？**
A: 这个版本优化了中文简历。如果需要英文简历，可以提供一个增强版本，支持语言切换。

**Q: 简历可以导出为哪些格式？**
A: 默认同时输出 HTML 和 DOCX 两种格式。HTML 还可以通过浏览器的"打印为 PDF"功能导出 PDF。

**Q: 语言能力放在哪里展示？**
A: 语言能力融入"专业技能区块"中，与 AI 能力、技术基础、产品能力并列展示，不单独成模块。
