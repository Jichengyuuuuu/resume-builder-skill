# SKILL: resume-builder

帮助用户基于任意背景材料一次性生成专业中文简历（HTML + DOCX 双格式）。核心流程：**信息收集** → **简历生成** → **高阶顾问建议**。

---

## 1) Overview

### 触发方式（Triggers）

- "帮我生成简历" / "写简历" / "制作简历"
- "帮我改进这份简历，我要投 XX 岗位"
- "我要改投一个岗位"
- 任何涉及创建、编辑、改进或优化简历的请求

### 输入形式（接受任意组合）

- ✅ 现有简历文件（上传或粘贴内容）
- ✅ 目标岗位 JD
- ✅ 口头描述 / 工作流水账 / 项目文档
- ✅ 以上任意组合，**无需反复整理，统统扔给 Skill**

### 核心输出（Deliverables）

- **HTML 简历**：可在浏览器直接预览，支持打印为 PDF
- **DOCX 简历**：可用 Word / WPS 直接编辑投递
- **高阶顾问建议**：3-5 条精准求职建议（随简历自动输出）

### 质量标准

- 内容：成就优先，量化指标突出，无流水账
- 格式：ATS 友好，关键词对齐 JD
- 页数：默认一页，按工作年限动态判断
- 水印：**严禁添加任何生成工具水印**

## 2) 第一阶段：信息收集 (Context Gathering)

信息收集应**柔和而全面**，不强制用户提供所有字段。用户未提供的字段使用占位符呈现。

### 收集流程

1. **初始检测**：询问用户有哪些现成资料（旧简历 / JD / 口头描述）
2. **解析已有资料**：如用户上传了简历，先解析，再按结构补充
3. **分块补充**：按「基本信息 → 教育背景 → 工作经历 → 技能」顺序逐步收集
4. **质量门控**：生成前确认至少有：姓名、工作经历、教育背景

### 需收集的字段

**个人基本信息**
- 姓名、电话、邮箱、城市/地区
- 个人网站 / GitHub / 博客（可选）
- 工作年限

**职业目标**（可选但推荐）
- 期望职位名称、职业方向描述、期望行业或公司类型

**教育背景**
- 学校名称、专业/学位、入学/毕业年份
- GPA（可选）、主修课程或获奖（可选）

**工作经历**（每段）
- 公司名称、职位名称、工作周期（开始-结束）、工作地点（可选）
- 工作描述和成就（**重点收集量化指标**，如"提升 30% 性能"、"服务 1000万+ 用户"）

**项目经验**（可选）
- 项目名称、角色、时间、描述与贡献（含量化指标）、技术栈

**专业技能**（见下方岗位定制规则）

**认证和荣誉**（可选）、**其他信息**（可选）

### 技能模块：先分析岗位，再决定维度

⚠️ **不套固定模板，按目标岗位动态决定技能维度。**

**第一步：判断目标岗位类型**

| 岗位类型 | 核心技能维度（示例）|
|---------|---------------------|
| 后端工程师 | 编程语言 / 框架与中间件 / 数据库 / 分布式与高并发 / DevOps & 云 |
| AI 产品经理 | AI/LLM 能力 / 产品方法论 / 数据能力 / 行业认知 / 工具协作 |
| C 端运营 | 用户增长方法论 / 内容策划 / 数据分析 / 平台规则与玩法 / 工具栈 |
| 视觉设计师 | 设计能力（品牌/插画/动效）/ 设计工具（Figma/C4D/AE）/ 作品集 |
| 市场营销 | 品牌策略 / 内容营销 / 投放与渠道 / 数据归因 / 工具栈 |
| HRBP | HR 专业能力（招聘/OD/绩效/薪酬）/ 业务理解 / 数据敏感度 |
| 财务分析师 | 财务专业能力 / 建模与估值 / 数据工具（Excel/SQL/BI）/ 行业理解 |

支持 11+ 种岗位类型，覆盖技术、产品、设计、运营、市场、销售、HR、财务、咨询、学术等。

**第二步：技能呈现通用规则**

- **避免过于基础的描述**："熟练使用 Office"、"会用电脑"是反向加分项
- **向上拔一层**："会做 PPT" → "数据可视化与商业演示"
- **匹配 JD 关键词**：如用户提供了 JD，技能关键词主动对齐
- **AI 关键词按需添加**：只在岗位相关时补充（AI/产品/运营/数据岗有效；财务/法务/行政不硬塞）
- **语言能力融入此区块**，不单独成模块

**第三步：技能不足时主动询问**

- 设计岗 → 问作品集、擅长风格、使用工具
- 运营岗 → 问活动量级、使用工具栈
- 销售岗 → 问最大单金额、客户类型、销售方法论

## 3) 第二阶段：简历生成 (Resume Generation)

收集足够信息后，**同时生成 HTML 和 DOCX 两种格式**。

### 设计规范

- **配色**：主文本深灰 #333333，辅助文本 #666666，背景纯白 #FFFFFF，重点色深蓝 #003366
- **字体**：-apple-system, BlinkMacSystemFont, Segoe UI, 微软雅黑
- **布局**：单列或双列，清晰视觉层级，充分留白
- **可打印性**：A4 纸张，@media print 适配

### 页数控制（重要）

| 工作年限 | 页数目标 |
|---------|---------|
| ≤ 5 年 | 强烈建议一页 |
| 6-10 年 | 尽量一页，放不下允许两页 |
| 10 年以上 | 两页可接受，不超过两页 |

**压缩优先级**：先删减早期/相关性低的分点 → 再缩减字体/行距 → 最后考虑双列布局

- 每段工作经历控制在 **3 条**分点为主，最多不超过 5 条
- 早期工作经历（超过 8 年前）只保留公司、职位、时间，不展开分点
- 超出一页时：不强行压缩，自然延伸，但**告知用户当前页数及建议删减方向**

### 简历结构

**1. 头部信息区块**
```
[姓名 - 大号、加粗]
[职位/求职方向] | [城市] | [电话] | [邮箱] | [GitHub/个人网站]
[职业总结 - 1-2 句话]
```

**2. 专业技能区块**
- 按岗位类型动态确定维度（参考上方对照表）
- tag 或紧凑行内格式，不占用过多纵向空间

**3. 工作经历区块**

每条分点采用 **"加粗概括短语 + 冒号 + 详细描述"** 格式：

```
[公司名称] - [职位] | [时间范围]
• **系统架构设计：**主导设计高并发分布式支付系统，日均处理交易 100万+，可用性 99.99%
• **性能优化：**优化核心查询链路，响应时间从 500ms 降至 50ms，提升 10 倍
• **团队管理：**带领 5 人小组完成 Q3 目标，提前 2 周交付
```

- ✅ `**数据治理：**建立企业级数据质量监控体系，覆盖 200+ 核心指标`
- ❌ `负责后端系统开发`（太笼统，缺少加粗概括）

**4. 项目经验区块**（如有）
```
[项目名称] | [角色] | [时间]
• **项目背景：**...
• **核心贡献：**...
• **技术栈：**...
```

**5. 教育背景区块**
```
[学校名称] - [学位/专业] | [时间]
GPA: X.X / 4.0（如有）
```

**6. 认证和荣誉区块**（如有）

### 优化策略

1. **ATS 优化**：使用标准职位名称，包含行业关键词，避免特殊符号
2. **成就优先**：动词 + 具体行为 + 量化结果；前 3 段工作经历最详细
3. **针对性优化**：有 JD 时优先突出相关技能和经历
4. **技能专业化**：口语化技能向专业方向提升，按岗位定制

⚠️ **禁止在简历任何位置添加生成水印**（包括"此简历由 XX 生成"等字样）

## 4) 双格式输出实施

### HTML 版本

- 使用 HTML5 + CSS3，内联 `<style>` 标签保证独立性
- 支持打印样式（@media print）
- 考虑响应式设计，手机可查看

### DOCX 版本

使用 `docx` npm 包生成。参考 `/mnt/skills/public/docx/SKILL.md` 获取详细用法。

关键要点：
- `npm install -g docx`
- A4 纸张尺寸（11906 x 16838 DXA）
- 字体：Arial / 微软雅黑
- 加粗概括短语用 `bold: true` 的 TextRun，冒号后用普通 TextRun
- 使用 `LevelFormat.BULLET` 配置列表，不要手动插入 unicode bullets
- 用 `Packer.toBuffer` 写入文件
- 生成后用 `python scripts/office/validate.py` 验证

```javascript
const { Document, Packer, Paragraph, TextRun, LevelFormat } = require('docx');

new Paragraph({
  numbering: { reference: "bullets", level: 0 },
  children: [
    new TextRun({ text: "系统架构设计：", bold: true, font: "Microsoft YaHei", size: 21 }),
    new TextRun({ text: "主导设计高并发分布式支付系统，日均处理交易 100万+", font: "Microsoft YaHei", size: 21 }),
  ]
})
```

### 输出流程

1. 在 `/home/claude` 下生成 HTML 和 DOCX 文件
2. 验证 DOCX：`python scripts/office/validate.py resume.docx`
3. 复制两个文件到 `/mnt/user-data/outputs/`
4. 使用 `present_files` 工具同时展示两个文件

## 5) 第三阶段：高阶顾问建议 (Senior Advisor Feedback)

简历生成并交付后，**自动**以比用户当前岗位高 1-2 级的视角（如用户是 PM 则以资深产品总监视角）给出建议。**无需用户额外请求。**

### 核心原则

- **角色代入**：以"如果我是面试你目标岗位的 hiring manager"身份说话
- **精准不啰嗦**：每次只挑 3-5 条最关键的建议
- **按需选择维度**：不需要每次都覆盖所有维度，聚焦实际存在的短板

### 可选建议维度

| 维度 | 说明 | 示例 |
|------|------|------|
| 信息补充 | 指出缺失的关键信息 | "提到了数据看板但未说明数据规模和技术方案" |
| 求职方向 | 判断目标岗位匹配度，有 gap 直接说 | "经历偏效率工具，投商业化 PM 需补充营收认知" |
| 亮点提炼 | 挖掘用户未意识到的亮点 | "从纸质到线上是 change management 的故事" |
| 面试准备 | 预判 2-3 个高频考点，给回答框架 | "B 端商业化岗必问如何衡量产品商业价值" |
| 风险点 | 面试官可能 challenge 的弱点 | "只有一段经历，准备好'为何深耕 3 年'的故事线" |

### 输出格式（直接写在对话中，不生成文件）

```
💡 高阶顾问建议（以资深产品总监视角）

1. **补充商业化认知：**你的经历集中在效率工具侧，但目标岗位是"商业化 PM"，建议...
2. **亮点提炼 — 变革推动力：**从纸质到线上不只是产品设计，更是组织变革...
3. **面试高频考点：**B 端商业化岗大概率会问...
```

### 约束

- 不重复简历已有内容
- 不写通用求职鸡汤，每条必须锚定用户具体情况
- 不超过 5 条
- 语气直接、平等，像愿意帮忙的前辈

## 7) 常见问题

**Q: 用户没有提供某些信息怎么办？**
A: 使用占位符（如 `[你的姓名]`），可选字段在用户明确后再补充。

**Q: 支持英文简历吗？**
A: 当前版本优化了中文简历。英文简历可使用此 Skill，但部分提示词针对中文场景。

**Q: 可以导出哪些格式？**
A: 默认同时输出 HTML + DOCX。HTML 可通过浏览器打印为 PDF。

**Q: 语言能力放在哪里？**
A: 融入专业技能区块，不单独成模块。
